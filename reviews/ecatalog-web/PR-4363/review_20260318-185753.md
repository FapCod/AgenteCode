## 🟡 Problemas Importantes (P1 - High Priority)
> *Problemas de performance, bugs probables o deuda técnica alta.*

1. **`apiError.statusCode` es `undefined` en el log (Bug Lógico)**
   - **Archivo**: `src/graphql/server/dataSources/ConsultantAPI.js` líneas ~44-54
   - **Problema**: El log de `saveLog` se invoca con `status: apiError.statusCode` y `body: apiError.originalError`, pero ambas propiedades se asignan en las líneas SIGUIENTES al bloque del log. Los campos `status` y `body` siempre serán `undefined` en Kibana.
   - **Impacto**: Los logs de error en Kibana carecen del código HTTP de estado y del error original, haciendo el diagnóstico de incidentes significativamente más difícil.
   - **Impacto en score**: -8 puntos (Funcionalidad, Robustez)
   - **Tiempo est.**: 10 minutos
   - **Categoria afectada**: Funcionalidad / Robustez
   - **Código actual**:

     ```js
     const apiError = new Error(
       `Error fetching consultant from API: ${error.message || 'Unknown error'}`
     );

     if (shouldRegisterLogKibana) {
       await errorLogger.saveLog(apiError, {
         countryCode,
         params,
         error: apiError.message,
         status: apiError.statusCode,    // ← UNDEFINED aquí
         body: apiError.originalError    // ← UNDEFINED aquí
       });
     }

     apiError.statusCode = error.extensions?.response?.status || 500; // ← asignado DESPUÉS
     apiError.originalError = error;                                   // ← asignado DESPUÉS
     throw apiError;
     ```

   - **Solución sugerida**: Mover las asignaciones de `statusCode` y `originalError` antes del bloque de log:

     ```js
     const apiError = new Error(
       `Error fetching consultant from API: ${error.message || 'Unknown error'}`
     );
     apiError.statusCode = error.extensions?.response?.status || 500;
     apiError.originalError = error;

     if (shouldRegisterLogKibana) {
       await errorLogger.saveLog(apiError, {
         countryCode,
         params,
         error: apiError.message,
         status: apiError.statusCode,    // ✅ ya tiene valor
         body: apiError.originalError    // ✅ ya tiene valor
       });
     }

     throw apiError;
     ```

2. **`sendToElastic` sin tolerancia a fallos (Dependencia Dura)**
   - **Archivo**: `src/graphql/server/dataSources/ConsultantAPI.js` líneas ~78 y ~100
   - **Problema**: Las llamadas a `errorLogger.sendToElastic(context)` y `errorLogger.saveLog(apiError, ...)` no están envueltas en `try-catch`. Si Elasticsearch no está disponible (timeout, red caída), la excepción se propagará al flujo principal del consultor.
   - **Impacto**: Un fallo de Kibana/Elastic puede bloquear la apertura del catálogo digital para la consultora. El logging es una dependencia secundaria ("best-effort") y no debe romper el flujo de negocio.
   - **Impacto en score**: -10 puntos (Robustez, Escalabilidad)
   - **Tiempo est.**: 15 minutos
   - **Categoria afectada**: Robustez / Escalabilidad
   - **Código actual**:

     ```js
     // Línea ~78 — sin protección
     await errorLogger.sendToElastic(context);

     // Línea ~100 — sin protección
     await errorLogger.sendToElastic(context);

     // Línea ~45 — sin protección
     await errorLogger.saveLog(apiError, { ... });
     ```

   - **Solución sugerida**: Envolver cada llamada al logger en `try-catch` independiente:

     ```js
     try {
       await errorLogger.sendToElastic(context);
     } catch (logError) {
       console.error('Error al enviar log a Elastic:', logError);
     }
     ```

     Aplicar el mismo patrón para `errorLogger.saveLog(...)`.

3. **Llamadas secuenciales a Feature Flags (Waterfall Anti-pattern)**
   - **Archivo**: `src/graphql/server/dataSources/ConsultantAPI.js` líneas ~25-31
   - **Problema**: Dos llamadas `await getFeatureFlag()` ejecutadas en serie cuando no dependen la una de la otra. Se acumula la latencia de ambas peticiones de forma innecesaria.
   - **Impacto**: Latencia adicional en cada invocación a `getConsultantByCodeOrDocument`. Si cada flag tarda ~20ms, esto suma 40ms en lugar de 20ms por petición.
   - **Impacto en score**: -5 puntos (Escalabilidad)
   - **Tiempo est.**: 10 minutos
   - **Categoria afectada**: Escalabilidad
   - **Código actual**:

     ```js
     const featureFlagValidateSlugFormat = await featureFlagAPIV2.getFeatureFlag(
       'feature_validate_slug_format'
     );

     const featureFlagRegisterLogKibana = await featureFlagAPIV2.getFeatureFlag(
       'feature_register_log_kibana'
     );
     ```

   - **Solución sugerida**: Paralelizar con `Promise.all`:

     ```js
     const [featureFlagValidateSlugFormat, featureFlagRegisterLogKibana] = await Promise.all([
       featureFlagAPIV2.getFeatureFlag('feature_validate_slug_format'),
       featureFlagAPIV2.getFeatureFlag('feature_register_log_kibana')
     ]);
     ```

## 🎓 Aprendizajes Clave
- ❌ **Evitar**: Leer propiedades de un objeto antes de asignarlas — `apiError.statusCode` era `undefined` al momento del log.
- ✅ **Preferir**: Inicializar completamente un objeto de error antes de pasarlo a cualquier función.
- ❌ **Evitar**: Llamadas `await` secuenciales a operaciones independientes — suman latencias innecesariamente.
- ✅ **Preferir**: `Promise.all([...])` para operaciones paralelas independientes.
- ❌ **Evitar**: Dependencias externas de logging sin catch — si Kibana cae, el flujo de negocio no debe fallar.
- ✅ **Preferir**: Try-catch alrededor de operaciones de logging (best-effort).
