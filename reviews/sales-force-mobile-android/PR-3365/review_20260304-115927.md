## 🟢 Recomendaciones (P2/P3 - Medium/Low)
> *Mejoras de mantenibilidad y buenas prácticas. (Ver Regla 3.3: NO sugerir comentarios/docs/tests extra)*

**P2 - Medium Priority:**
- 💡 Hay una ligera duplicación de lógica entre los distintos `PersonalDataScreenViewModelXX` para obtener el tipo de documento y el número de documento (por ejemplo, normalización de formato con `replace` o acceso a `fieldsMap`). Si en el futuro se amplía el número de países o se cambia la forma de representar documentos, podría ser útil concentrar esta lógica en utilidades o en funciones compartidas de dominio para reducir la probabilidad de inconsistencias entre mercados.
- **Solución sugerida**:
     ```kotlin
     fun DocumentNumberNormalizer.normalize(docType: DocumentType, rawValue: String): String {
         return when (docType) {
             DocumentTypesConstants.RUT -> rawValue.replace(Regex("[.-]"), "")
             DocumentTypesConstants.DPI -> rawValue.replace(" ", "")
             // ...
             else -> rawValue
         }
     }
     ```

**P3 - Low Priority:**
- ℹ️ En algunos viewmodels se usa directamente `toString()` sobre `docType` para obtener el valor que se envía a backend. Asegúrate de que esto sea consistente con el contrato de la API (por ejemplo, si en el futuro se cambia el `toString()` de `DocumentType`, podría ser preferible usar una propiedad explícita como `code` o `id` si existe).
