## 🔴 Problemas Críticos (P0 - Blocking)

> *Issues de seguridad, crashes, o violaciones de reglas estrictas. Deben corregirse obligatoriamente.*

1. **Deadlock Potencial: `.GetAwaiter().GetResult()` en Método Síncrono**
   - **Archivo**: `MenuLogic.cs` línea 76
   - **Problema**: Se usa `Task.Run(() => GetComplaintsBook(UserData.PaisID)).GetAwaiter().GetResult()` para llamar un método `async` desde un contexto síncrono. En ASP.NET Classic (no Core), esto puede causar deadlock porque el `SynchronizationContext` del hilo de la petición HTTP queda bloqueado esperando la tarea.
   - **Riesgo**: Deadlock en producción bajo carga, congelando peticiones HTTP y agotando el ThreadPool.
   - **Impacto en score**: -2 pts (Robustez, Arquitectura)
   - **Tiempo est.**: 30 minutos
   - **Categoria afectada**: Robustez / Arquitectura
   - **Código actual**:

     ```csharp
     public MenuSchema GetAllowedMenuItems(dynamic ViewBag)
     {
         var complaintsBook = Task.Run(() => GetComplaintsBook(UserData.PaisID)).GetAwaiter().GetResult();
     ```

   - **Solución sugerida**: Hacer `GetAllowedMenuItems` async o refactorizar la cadena de llamadas. La solución más limpia es propagar `async/await` hacia arriba:

     ```csharp
     public async Task<MenuSchema> GetAllowedMenuItemsAsync(dynamic ViewBag)
     {
         var complaintsBook = await GetComplaintsBook(UserData.PaisID);
         // ...
     }
     ```

     Si el método debe permanecer síncrono estrictamente, ejecutar la lógica de `GetComplaintsBook` de forma síncrona directamente (sin `async/await`) para evitar el bloqueo sobre el SynchronizationContext.

## 🟡 Problemas Importantes (P1 - High Priority)

> *Problemas de performance, bugs probables o deuda técnica alta.*

1. **Instanciación Directa (`new TablaLogicaProvider()`) Rompe la Inyección de Dependencias**
   - **Archivo**: `MenuLogic.cs` línea 69
   - **Problema**: `_tablaLogicaProvider` se instancia directamente con `new TablaLogicaProvider()` en lugar de recibirlo por inyección de dependencias. Esto acopla fuertemente `MenuLogic` a la implementación concreta, dificulta el mockeado para pruebas y evita que el contenedor gestione el ciclo de vida.
   - **Impacto**: Dificulta testabilidad y rompe el patrón de DI establecido en el proyecto.
   - **Impacto en score**: -1 pt (Arquitectura, Testabilidad)
   - **Tiempo est.**: 20 minutos
   - **Categoria afectada**: Arquitectura
   - **Código actual**:

     ```csharp
     public MenuLogic()
     {
         SessionManager = Web.SessionManager.SessionManager.Instance;
         UserData = SessionManager.GetUserData();
         _tablaLogicaProvider = new TablaLogicaProvider();
     }
     ```

   - **Solución sugerida**: Si la clase `MenuLogic` soporta constructor con parámetros, inyectar via constructor:

     ```csharp
     public MenuLogic(ITablaLogicaProvider tablaLogicaProvider)
     {
         SessionManager = Web.SessionManager.SessionManager.Instance;
         UserData = SessionManager.GetUserData();
         _tablaLogicaProvider = tablaLogicaProvider;
     }
     ```

2. **Acceso a `item.SubMenu` sin Validación de Null**
   - **Archivo**: `MenuLogic.cs` línea 120
   - **Problema**: `item.SubMenu.Insert(...)` se llama sin verificar si `item.SubMenu` es null. Si algún elemento del menú no tiene submenú inicializado, provocará un `NullReferenceException` en producción.
   - **Impacto**: Crash en producción si algún ítem del menú móvil no tiene SubMenu.
   - **Impacto en score**: -1 pt (Robustez)
   - **Tiempo est.**: 10 minutos
   - **Categoria afectada**: Robustez
   - **Código actual**:

     ```csharp
     if (item.MenuMobileID == Constantes.LibroReclamaciones.SeccionLegalesId)
     {
         item.SubMenu.Insert(Constantes.LibroReclamaciones.Orden, new MenuMobileModel { ... });
     }
     ```

   - **Solución sugerida**:

     ```csharp
     if (item.MenuMobileID == Constantes.LibroReclamaciones.SeccionLegalesId)
     {
         item.SubMenu ??= new List<MenuMobileModel>();
         item.SubMenu.Insert(Constantes.LibroReclamaciones.Orden, new MenuMobileModel { ... });
     }
     ```

## 📋 Resumen del PR
**¿Qué hace este PR?**
Implementa la funcionalidad de "Libro de Reclamaciones" en el portal, exponiendo el link tanto en el menú móvil como en la sección Home. El backend (C#) carga la configuración desde `TablaLogica` y la expone a través del endpoint de menú; el frontend (React/TypeScript) consume esa data para renderizar el componente.

**Archivos principales modificados:**
- [NEW] `complaints-book/index.tsx` - Nuevo componente React del Libro de Reclamaciones
- [NEW] `complaints-book/style.ts` - Estilos del componente
- [MODIFY] `MenuLogic.cs` - Lógica principal de obtención y guardado en menú
- [MODIFY] `ConsTablaLogica.cs` y `Constantes.cs` - Nuevas constantes de configuración
- [MODIFY] `MenuSchema.cs` y `aspnet-webapi-schemas.ts` - Nuevos schemas para el schema de respuesta
- [NEW] `complaintsbook.inline.svg` - Ícono SVG del libro de reclamaciones

**¿Qué podria impactar?**
La modificación de `GetAllowedMenuItems` en `MenuLogic.cs` afecta a **todos los usuarios** que carguen el menú, ya que se agrega una llamada async bloqueante en el flujo principal de construcción del menú.

**Impacto:**
- PERFORMANCE: El bloqueo síncrono de una tarea async puede degradar el rendimiento bajo carga alta.
- ARQUITECTURA: La instanciación directa de `TablaLogicaProvider` rompe el patrón de DI del proyecto.
