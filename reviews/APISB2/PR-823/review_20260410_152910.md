## 🟡 Problemas Importantes (P1 - High Priority)

1. **[Potencial NullReferenceException en GetConnectionName]**
   - **Archivo**: `app/Belcorp.Api.Common/SecretFactory/AwsSecretsManagerStrategy.cs` líneas 89-98
   - **Problema**: Si `LoadSecretsFromAws()` falla con excepción (que se loguea y re-lanza), `_connectionStringsCache` permanece null. En `GetConnectionName()` se accede a `_connectionStringsCache.TryGetValue()` sin validar null.
   - **Impacto**: Si AWS no está disponible, la aplicación crasheará al intentar obtener cualquier setting.
   - **Impacto en score**: -4 puntos
   - **Tiempo est.**: 5 minutos
   - **Categoria afectada**: Robustez
   - **Código actual**:

     ```csharp
     public string GetConnectionName(string connectionKeyName)
     {
         EnsureCacheUpToDate();

         if (_connectionStringsCache.TryGetValue(connectionKeyName, out string value))
         {
             return value;
         }
         return null;
     }
     ```

   - **Solución sugerida**: Agregar validación de null

     ```csharp
     public string GetConnectionName(string connectionKeyName)
     {
         EnsureCacheUpToDate();

         if (_connectionStringsCache == null)
         {
             return null;
         }

         if (_connectionStringsCache.TryGetValue(connectionKeyName, out string value))
         {
             return value;
         }
         return null;
     }
     ```

2. **[Cambio de comportamiento en valores de retorno - Breaking Change]**
   - **Archivo**: `app/Belcorp.Api.Common/WebConfig.cs` múltiples propiedades
   - **Problema**: Las propiedades ahora retornan `null` cuando antes retornaban `""` (string vacío). Esto puede causar `NullReferenceException` en código que asume strings no-nulos.
   - **Impacto**: Código existente que hacía `WebConfig.MandrillKey.Length` o similar crasheará.
   - **Impacto en score**: -4 puntos
   - **Tiempo est.**: 10 minutos
   - **Categoria afectada**: Robustez
   - **Código actual**:

     ```csharp
     public static string MandrillKey
     {
         get { return AppSettingsOrchestrator.GetAppSetting("MandrillKey"); }
     }
     ```

   - **Solución sugerida**: Mantener el comportamiento original con null coalescing

     ```csharp
     public static string MandrillKey
     {
         get { return AppSettingsOrchestrator.GetAppSetting("MandrillKey") ?? ""; }
     }
     ```

3. **[Static initializer puede fallar silenciosamente]**
   - **Archivo**: `app/Belcorp.Api.Common/SecretFactory/AppSettingsOrchestrator.cs` línea 8
   - **Problema**: El static readonly field `_strategy` se inicializa cuando se accede la clase por primera vez. Si AWS falla, lanzará `TypeInitializationException` que es difícil de debuggear.
   - **Impacto**: Crash de la aplicación con excepción poco descriptiva.
   - **Impacto en score**: -2 puntos
   - **Tiempo est.**: 15 minutos
   - **Categoria afectada**: Robustez
   - **Código actual**:

     ```csharp
     public static class AppSettingsOrchestrator
     {
         private static readonly AwsSecretsManagerStrategy _strategy = 
             new AwsSecretsManagerStrategy(awsSecretsName.APP_SETTINGS_SECRET_NAME);
     ```

   - **Solución sugerida**: Usar Lazy<T> para manejo de errores más controlado o try-catch con fallback

     ```csharp
     public static class AppSettingsOrchestrator
     {
         private static readonly Lazy<AwsSecretsManagerStrategy> _strategy = 
             new Lazy<AwsSecretsManagerStrategy>(() => 
             {
                 try
                 {
                     return new AwsSecretsManagerStrategy(awsSecretsName.APP_SETTINGS_SECRET_NAME);
                 }
                 catch (Exception ex)
                 {
                     LogManager.SaveLog(ex, "Error inicializando AWS Secrets");
                     return null;
                 }
             });

         public static string GetAppSetting(string key)
         {
             if (_strategy.Value == null)
             {
                 return ConfigurationManager.AppSettings[key];
             }
             return _strategy.Value.GetConnectionName(key);
         }
     }
     ```

## 📋 Resumen del PR

**¿Qué hace este PR?**
Migra la gestión de secretos desde hardcoded en archivos de configuración hacia AWS Secrets Manager. Implementa un sistema de caché con TTL configurable para los secretos y reemplaza todas las referencias a `ConfigurationManager.AppSettings["KEY"]` por `AppSettingsOrchestrator.GetAppSetting("KEY")`.

**Archivos principales modificados:**
- [NEW] `app/Belcorp.Api.Common/SecretFactory/AppSettingsOrchestrator.cs` - Orquestador de settings
- [NEW] `app/Belcorp.Api.Common/SecretFactory/AwsSecretsManagerStrategy.cs` - Cliente AWS con caché
- [NEW] `app/Belcorp.Api.Common/SecretFactory/IConnectionStringStrategy.cs` - Interface Strategy
- [NEW] `app/Belcorp.Api.Common/SecretFactory/SecretsBootstrap.cs` - Bootstrap de connection strings
- [MODIFY] `app/Belcorp.Api/Web.config` - Remoción de secretos, placeholders
- [MODIFY] `app/Belcorp.Api.WindowsService/App.config` - Remoción de secretos
- [MODIFY] `app/Belcorp.Api.Common/WebConfig.cs` - Migración a AppSettingsOrchestrator
- [MODIFY] Múltiples archivos en ServicioExterno/ - Migración a AppSettingsOrchestrator

**¿Qué podría impactar?**
- Conexiones a bases de datos de todos los países (15 connection strings)
- Servicios de buró de crédito (Equifax, Compliance, CirculoCredito, etc.)
- Autenticación JWT
- Integración con AWS S3 y DynamoDB
- Servicios de email (Mandrill, SMTP)

**Impacto:**
- **Seguridad**: POSITIVO CRÍTICO - Elimina exposición de secretos
- **Arquitectura**: POSITIVO - Centraliza gestión de secretos
- **Performance**: NEUTRAL - Caché evita llamadas excesivas a AWS
