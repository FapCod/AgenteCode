## 🟡 Problemas Importantes (P1 - High Priority)

> *Problemas de performance, bugs probables o deuda tecnica alta.*

1. **[Inconsistencia de Logging - console.error sin logger centralizado]**
   - **Archivo**: Multiples archivos
   - **Problema**: El proyecto usa `ctx.log` (koa-pino-logger) como logger centralizado, pero varios archivos nuevos usan `console.error` directamente.
   - **Impacto**: Perdida de features de logging centralizado (formato, niveles, contexto, integracion con Kibana/ElasticSearch).
   - **Impacto en score**: -6 puntos
   - **Tiempo est.**: 15 minutos
   - **Categoria afectada**: Robustez, Mantenibilidad
   - **Archivos afectados**:
     - `api-auth/newrelic.js`
     - `api-auth/src/controllers/Auth.js` (metodo `tokenExchange`)
     - `api-auth/src/utils/azureAuth.js`
   - **Codigo actual**:

     ```javascript
     // En Auth.js linea 111
     console.error("OAuth Exchange Failed:", error.message);
     
     // En azureAuth.js lineas 19, 29
     console.error("MICROSOFT_GRAPH URL not configured");
     console.error("Graph Validation Failed:", status);
     ```

   - **Solucion sugerida**: Para archivos donde hay contexto Koa disponible, usar `ctx.log.error()`. Para utilidades donde no hay contexto disponible (como `env.js`, `newrelic.js`), considerar usar pino directamente o crear un logger singleton.

     ```javascript
     // Opcion A: En controladores con ctx disponible
     ctx.log.error({ error: error.message }, "OAuth Exchange Failed");
     
     // Opcion B: Para utilidades sin ctx, crear logger singleton
     import pino from 'pino';
     const logger = pino();
     logger.error("MICROSOFT_GRAPH URL not configured");
     ```

   > **Nota**: Los `console.error` en `env.js` y `env.ts` para errores de parseo de `APP_CONFIG` son aceptables dado que ocurren en tiempo de inicializacion antes de que el sistema de logging este disponible.

   - **Problema**: Se usa `console.warn` para informar que se usa yenv por defecto. Este mensaje aparecera en cada inicio de la aplicacion en desarrollo.

## 📋 Resumen del PR

**¿Que hace este PR?**
Este PR migra la configuracion de la aplicacion desde archivos `env.yaml` estaticos hacia variables de entorno (`APP_CONFIG`). Esto elimina credenciales hardcodeadas del repositorio, mejorando significativamente la seguridad. Ademas, agrega un nuevo endpoint `/token-exchange` para intercambio de tokens Azure AD y funcionalidad de encriptacion de secrets.

**Archivos principales modificados:**
- [DELETE] `api-auth/env.yaml` - Eliminacion de credenciales hardcodeadas
- [DELETE] `providers/adfs-auth/env.yaml` - Eliminacion de credenciales
- [DELETE] `providers/basic-auth/env.yaml` - Eliminacion de credenciales
- [NEW] `api-auth/src/utils/env.js` - Nueva utilidad de configuracion
- [NEW] `api-auth/src/utils/azureAuth.js` - Validacion con Microsoft Graph
- [NEW] `api-auth/src/utils/encrypt-secrets.js` - Encriptacion de secrets
- [NEW] `api-auth/src/helpers/encryptionHelper.js` - Helper de encriptacion AES-256-GCM
- [MODIFY] `api-auth/src/controllers/Auth.js` - Nuevo endpoint tokenExchange
- [MODIFY] `api-auth/src/routes/auth.js` - Nueva ruta /token-exchange
- [MODIFY] Multiples archivos - Migracion de `yenv()` a `getEnv()`

**¿Que Podria impactar?**
- **Despliegue**: Requiere que la variable de entorno `APP_CONFIG` este configurada correctamente en todos los ambientes (QAS, PPR, PRD).
- **Dependencias**: Si `APP_CONFIG` no esta presente, el sistema fallback a yenv (desarrollo local).
- **Integraciones**: El nuevo endpoint `/token-exchange` afecta flujos de autenticacion con Azure AD.

**Impacto:**
- SEGURIDAD: Mejora significativa al eliminar credenciales del codigo fuente
- ARQUITECTURA: Cambio en la forma de obtener configuracion en toda la aplicacion
