## 🟡 Problemas Importantes (P1 - High Priority)
> *Problemas de performance, bugs probables o deuda técnica alta.*

1. **[P1] Números mágicos para el timeout de procesos pendientes**
   - **Archivo**: `app/Belcorp.ProcesoAutomatico.BizLogic/PagoEnLinea/BLPagoEnLinea.cs` línea 45 (aprox.)
   - **Problema**: El umbral de 30 minutos para cancelar automáticamente un proceso en estado `Pendiente` está codificado como número mágico (`30`) dentro de la expresión de comparación de tiempo. Esto dificulta ajustar la regla de negocio y reduce la auto-documentación de la intención.
   - **Impacto**: Deuda técnica y mayor riesgo de inconsistencias si en el futuro otro módulo necesita usar el mismo umbral o se decide modificarlo.
   - **Tiempo est.**: 10-15 minutos
   - **Código actual**:
     ```csharp
     if ((DateTime.Now - procesoAutomatico.FechaInicio).TotalMinutes >= 30)
     {
         procesoAutomatico.Estado = ProcesoEstadoEnum.Error;
         procesoAutomatico.ErrorLog = "Cancelado automáticamente por timeout tras quedar en estado Pendiente por 30 minutos o más.";
         procesoAutomatico.ErrorProceso = "Actualizar Proceso Automatico";
         try
         {
             daProcesoAutomatico.Update(codigoIso, procesoAutomatico);
         }
         catch (Exception ex)
         {
             LogManager.SaveProcessException(ex, codigoIso, CodigoProceso, procesoAutomatico.ProcesoAutomaticoId);
         }
     }
     ```
   - **Solución sugerida**:
     ```csharp
     private const int PendingProcessTimeoutMinutes = 30;
     
     // ...
     if ((DateTime.Now - procesoAutomatico.FechaInicio).TotalMinutes >= PendingProcessTimeoutMinutes)
     {
         procesoAutomatico.Estado = ProcesoEstadoEnum.Error;
         procesoAutomatico.ErrorLog = "Cancelado automáticamente por timeout tras quedar en estado Pendiente por 30 minutos o más.";
         procesoAutomatico.ErrorProceso = "Actualizar Proceso Automatico";
         try
         {
             daProcesoAutomatico.Update(codigoIso, procesoAutomatico);
         }
         catch (Exception ex)
         {
             LogManager.SaveProcessException(ex, codigoIso, CodigoProceso, procesoAutomatico.ProcesoAutomaticoId);
         }
     }
     ```

2. **[P1] Valor fijo de CommandTimeout repetido en acceso a datos**
   - **Archivo**: `app/Belcorp.ProcesoAutomatico.Data/DAProcesoAutomatico.cs` líneas 20, 48 y 66 (aprox.)
   - **Problema**: El valor `300` de `CommandTimeout` se repite en las operaciones de lectura, inserción y actualización del proceso automático. Al ser un timeout crítico de base de datos, conviene centralizarlo en una constante o configuración para evitar divergencias en el futuro.
   - **Impacto**: Deuda técnica y mayor esfuerzo de mantenimiento si se requiere ajustar el timeout (riesgo de que algunas llamadas queden con un valor distinto por olvido).
   - **Tiempo est.**: 10-15 minutos
   - **Código actual**:
     ```csharp
     DbCommand dbCommand = database.GetStoredProcCommand("interfaces.GetLastProcesoAutomaticoByCodigoProceso", codigoProceso);
     dbCommand.CommandTimeout = 300;
     
     // ...
     DbCommand dbCommand = database.GetStoredProcCommand("interfaces.InsProcesoAutomatico", codigoProceso);
     dbCommand.CommandTimeout = 300;
     
     // ...
     oDbCommand.CommandTimeout = 300;
     oDatabase.ExecuteNonQuery(oDbCommand);
     ```
   - **Solución sugerida**:
     ```csharp
     private const int DefaultCommandTimeoutSeconds = 300;
     
     // ...
     DbCommand dbCommand = database.GetStoredProcCommand("interfaces.GetLastProcesoAutomaticoByCodigoProceso", codigoProceso);
     dbCommand.CommandTimeout = DefaultCommandTimeoutSeconds;
     
     // ...
     DbCommand dbCommand = database.GetStoredProcCommand("interfaces.InsProcesoAutomatico", codigoProceso);
     dbCommand.CommandTimeout = DefaultCommandTimeoutSeconds;
     
     // ...
     oDbCommand.CommandTimeout = DefaultCommandTimeoutSeconds;
     oDatabase.ExecuteNonQuery(oDbCommand);
     ```

## 🟢 Recomendaciones (P2/P3 - Medium/Low)
> *Mejoras de mantenibilidad y buenas prácticas. (Ver Regla 3.3: NO sugerir comentarios/docs/tests extra)*

**P2 - Medium Priority:**
- 💡 Considerar parametrizar el umbral de minutos del timeout del proceso pendiente desde configuración (appSettings/feature flag) si en otros países o entornos se requiere un valor distinto al estándar de 30 minutos.
- **Solución sugerida**:
     ```csharp
     var pendingTimeoutMinutes = configuracion.TimeoutPendienteEnMinutos ?? 30;
     if ((DateTime.Now - procesoAutomatico.FechaInicio).TotalMinutes >= pendingTimeoutMinutes)
     {
         // ...
     }
     ```

- 💡 Evaluar si el timeout de 300 segundos para los stored procedures de `interfaces` debería estar alineado a una política global de timeouts (por ejemplo, una constante compartida para interfaces críticas o leída desde configuración central).
- **Solución sugerida**:
     ```csharp
     private readonly int _interfacesCommandTimeoutSeconds;
     
     public DAProcesoAutomatico(IConfiguration configuration)
     {
         _interfacesCommandTimeoutSeconds = configuration.GetValue<int?>("Interfaces:CommandTimeoutSeconds") ?? 300;
     }
     ```

**P3 - Low Priority:**
- ℹ️ Revisar que la cadena de error `"Cancelado automáticamente por timeout tras quedar en estado Pendiente por 30 minutos o más."` siga las convenciones de internacionalización del sistema (si existiera una capa de mensajes centralizada). A nivel de robustez está bien, pero podría trasladarse a un recurso compartido si se internacionalizan mensajes operativos.

## 📋 Resumen del PR
**¿Qué hace este PR?**
Introduce una lógica de timeout sobre procesos de pago en línea que permanecen en estado `Pendiente` por 30 minutos o más, marcándolos automáticamente como `Error` y registrando un mensaje claro, además de estandarizar el `CommandTimeout` de las operaciones de base de datos relacionadas con el proceso automático a 300 segundos.

**Archivos principales modificados:**
- [MODIFY] `app/Belcorp.ProcesoAutomatico.BizLogic/PagoEnLinea/BLPagoEnLinea.cs` - Agrega control de timeout para procesos en estado `Pendiente` y logging de error asociado.
- [MODIFY] `app/Belcorp.ProcesoAutomatico.Data/DAProcesoAutomatico.cs` - Establece un `CommandTimeout` explícito de 300 segundos para las operaciones de lectura, inserción y actualización de procesos automáticos.

**¿Qué Podria impactar?**
El flujo de ejecución de procesos de pago en línea que se quedaban pendientes, así como los tiempos de espera máximos al ejecutar procedimientos almacenados de `interfaces` para estos procesos.
